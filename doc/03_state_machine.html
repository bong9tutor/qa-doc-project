Server Authoritative 원칙: 클라이언트는 입력(Input)만 서버로 전송하고(ServerRPC_SetInputState), 서버가 상태를 계산하여 ReplicatedUsing 또는 Replicated로 모든 클라이언트에 전파합니다. 상태 변경은 항상 서버에서 시작되며, 클라이언트의 OnRep 콜백에서 UI 업데이트나 시각 효과를 처리합니다.
QA 버그 포인트: GameState의 GameStartTime과 GameEndTime이 OnRep_TeamDeathCount 콜백을 재사용하고 있습니다 (동일한 OnRep 함수를 3개 UPROPERTY가 공유). 이는 의도적으로 삽입된 버그 후보이며, 시간 값 변경 시 올바른 UI 갱신이 이루어지지 않을 수 있습니다. 
// AO_GameState.h — OnRep 공유 버그 후보
UPROPERTY(ReplicatedUsing = OnRep_TeamDeathCount)
int32 TeamDeathCount;

UPROPERTY(ReplicatedUsing = OnRep_TeamDeathCount)  // 동일한 OnRep!
float GameStartTime;

UPROPERTY(ReplicatedUsing = OnRep_TeamDeathCount)  // 동일한 OnRep!
float GameEndTime;
AI StateTree 상태 전이
모든 AI 행동은 UE5 StateTree로 구동됩니다. 서버에서만 실행되며, 결과는 복제를 통해 클라이언트에 반영됩니다.
StateTree 실행 구조
StateTree는 3가지 핵심 요소로 구성됩니다:

1. Evaluator — 매 틱마다 AI의 현재 상황을 평가하여 컨텍스트 데이터를 갱신합니다.
  예: FAO_STEval_AggressiveCtx가 시야 내 플레이어 존재, 추격 상태, 공격 범위, 스턴 여부 등을 평가

2. Condition — 상태 전이 조건을 판정합니다. TestCondition()이 true/false를 반환합니다.
  예: FAO_STCond_PlayerInSight (시야 체크), FAO_STCond_InAttackRange (공격 범위 체크)

3. Task — 실제 행동을 수행합니다. EnterState / Tick / ExitState 생명주기를 따릅니다.
  예: FAO_STTask_AIRoam (배회), FAO_STTask_Chase (추격), FAO_STTask_Bull_Charge (돌진)
Bull AI 상태 흐름 예시
Roam → PlayerDetected → Chase → InRange → Charge → Cooldown → Chase
STCond_PlayerInSight → STTask_Chase → STCond_InAttackRange → STTask_Bull_Charge → bInPostAttackCooldown → Loop
Roam: FAO_STTask_AIRoam — NavMesh 내 랜덤 위치로 배회 (반경 1000, 도착 후 2초 대기)
PlayerDetected: FAO_STCond_PlayerInSight — AI Perception으로 시야 내 플레이어 감지 시 전이
Chase: FAO_STTask_Chase — 타겟 플레이어 추격, 0.3초 간격 경로 재계산
InRange: FAO_STCond_InAttackRange — 추격 대상이 공격 범위 내에 진입했는지 판정
Charge: FAO_STTask_Bull_Charge — GAS 돌진 어빌리티 실행, 완료 대기
Cooldown: Evaluator의 bInPostAttackCooldown — 공격 후 쿨다운 대기, 이후 Chase로 복귀

Server-Only 실행: StateTree는 서버의 AIController에서만 Tick됩니다. AI 캐릭터의 이동과 애니메이션은 CharacterMovementComponent의 네트워크 복제를 통해 클라이언트에 자동 반영됩니다. GAS 어빌리티(공격 등)는 서버에서 활성화되고 Multicast RPC로 시각 효과를 전파합니다.
GAS 어빌리티 상태 연관
Gameplay Ability System(GAS)의 태그 기반 상태 관리가 캐릭터 상태 머신과 어떻게 연동되는지 설명합니다.
어빌리티 생명주기와 상태
GAS 어빌리티는 자체 상태 흐름을 가집니다:

CanActivate? → ActivateAbility → IsActive (실행 중) → EndAbility
CanActivateAbility() 체크 → 태그 요구사항 검증 → 실행 → CommitAbility → 종료
CanActivateAbility()는 다음을 검증합니다:
• Activation Required Tags — 캐릭터가 이 태그를 보유해야 활성화 가능
• Activation Blocked Tags — 캐릭터가 이 태그를 보유하면 활성화 차단
• Source/Target Required/Blocked Tags — 소스/타겟 조건
• 커스텀 조건 (스태미나 잔량 등)
Sprint 어빌리티 - 스태미나 연동 예시
// Sprint 어빌리티 활성화 조건
bool CanActivate():
  Stamina > 0                          // AttributeSet 체크
  !HasTag(State.Dead)                  // 사망 태그 없어야 함
  !HasTag(State.Stunned)               // 스턴 태그 없어야 함

// 실행 중 Tick
void TickAbility():
  Stamina -= DrainRate * DeltaTime     // 스태미나 소모
  if (Stamina <= 0)
    EndAbility()                       // 자동 종료 → Move 상태로 복귀
    Gait = EGait::Run                  // 서버에서 Gait 변경 → OnRep_Gait
Sprint 어빌리티가 종료되면 Gait이 Run으로 변경되며, OnRep_Gait()을 통해 모든 클라이언트의 애니메이션이 갱신됩니다.
Death 어빌리티 - 전체 차단
사망 어빌리티는 Blocking Tags를 사용하여 다른 모든 어빌리티의 활성화를 차단합니다:

// Death 어빌리티가 부여하는 태그
ActivationOwnedTags:
  State.Dead

// 다른 어빌리티들의 차단 조건
ActivationBlockedTags:
  State.Dead                    // 사망 중 활성화 불가

// Death 발동 흐름
HP AttributeSet → PostGameplayEffectExecute()
  → HP <= 0 감지
  → TryActivateAbilityByTag(Ability.Death)
  → Ragdoll ON, 입력 차단
  → PlayerState->SetIsAlive(false)  // 복제 → OnRep_IsAlive
  → 리스폰 타이머 시작
태그 기반 상태 관리 요약
GameplayTag부여 시점차단하는 어빌리티
State.Dead HP=0 → Death 어빌리티 활성화 Sprint, Attack, Interact, Crouch, Jump, Traversal 등 전체
State.Stunned AI 공격 피격 시 GameplayEffect 적용 Sprint, Attack, Interact, Jump
State.Interacting Interact 어빌리티 활성화 Sprint, Attack, Jump
Ability.Melee Attack 어빌리티 활성화 Sprint, Interact (활성화 중 중복 방지)
State.Traversal 파쿠르/트래버설 중 Attack, Interact, Sprint

상태 전이와 네트워크 복제
전이트리거복제 방법
Waiting → CountDown 모든 플레이어 Ready (bLobbyIsReady 전원 true) 서버 GameMode가 판정, ServerTravel로 맵 전환
CountDown → InProgress 카운트다운 0초 도달 서버 GameMode → AGameState::MatchState 복제
InProgress → Victory 승리 조건 달성 (연료 전량 수집 등) SetGameClear() → bIsGameCleared 복제 → OnRep_IsGameCleared
InProgress → Defeat 전원 사망 (SharedReviveCount == 0 AND 모든 bIsAlive == false) SetStageFailed() → bIsStageFailed 복제 → OnRep_IsStageFailed
Victory → RestRoom 결과 화면 종료 서버 ServerTravel → LV_RestRoom (인벤토리/HP를 PlayerState에 저장 후 이동)
RestRoom → CountDown 기차 출발 (다음 스테이지 선택) 서버 ServerTravel → 다음 스테이지 맵
Defeat → Waiting 패배 연출 종료 후 로비 복귀 서버 ServerTravel → LV_Lobby
레벨 이동 시 데이터 보존: PlayerState의 PersistentInventory와 PersistentHealth가 Replicated로 선언되어 있어, ServerTravel(Seamless Travel) 시 PlayerState가 유지되면서 인벤토리와 체력이 다음 스테이지로 이어집니다. CopyProperties() 오버라이드를 통해 PlayerState 간 데이터 이전을 처리합니다.
