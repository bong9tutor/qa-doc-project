AAO_Train
부모: AActor · IAbilitySystemInterface
역할
연료를 받아 움직이는 기차 액터 — GAS로 연료 수치 관리
연료 공급 방식
플레이어 인벤토리의 선택 슬롯에서 FuelAmount를 읽어 GameplayEvent로 전달
실행 위치
서버 Authority에서 GAS 어빌리티 Grant/Activate, ASC Minimal 복제 모드
실패 조건
연료가 0 미만으로 떨어지면 GameMode_Stage의 TriggerStageFailByTrainFuel() 호출
AActor와 IAbilitySystemInterface를 함께 상속하여 비-캐릭터 액터에도 GAS를 적용한 설계입니다. 생성자에서 StaticMesh, ASC, FuelAttributeSet, InteractableComp를 DefaultSubobject로 생성하고, InteractableComp의 OnInteractionSuccess 델리게이트에 HandleInteractionSuccess를 바인딩합니다. bReplicates = true이며 ASC는 Minimal 복제 모드로 설정되어 필수 어트리뷰트만 전파합니다.

BeginPlay에서 서버(HasAuthority)만 ASC->InitAbilityActorInfo(this, this)로 자기 자신을 Owner/Avatar로 설정하고, AddEnergy/LeakEnergy 두 어빌리티를 GiveAbility로 부여합니다. 그 후 모든 인스턴스(서버+클라이언트)에서 Fuel 어트리뷰트 변경 델리게이트에 OnFuelChanged를 바인딩합니다.

HandleInteractionSuccess는 Interactor의 UAO_InventoryComponent에서 선택 슬롯의 FuelAmount를 읽고, "Event.Interaction.AddFuel" GameplayTag로 FGameplayEventData를 구성하여 ASC->HandleGameplayEvent()로 전달합니다. 이벤트 기반으로 AddEnergy 어빌리티가 활성화되어 Fuel 어트리뷰트를 증가시킵니다. 성공 후 Inventory->ClearSlot()으로 소비된 아이템을 제거합니다.

연료 변경 콜백(OnFuelChanged)에서는 Delta(NewFuel - OldFuel)를 TotalFuelGained에 누적하고, 서버에서 OldFuel >= 0 && NewFuel < 0 조건이 성립하면 AAO_GameMode_Stage::TriggerStageFailByTrainFuel()로 스테이지 실패를 트리거합니다.

HandleInteractionSuccess(AActor* Interactor) ▶
구현 상세 (.cpp)
• HasAuthority() && ASC 유효성 체크 — 서버에서만 실행
• Interactor->FindComponentByClass<UAO_InventoryComponent>()로 인벤토리 획득
• Inventory->Slots[SelectedSlotIndex]에서 FInventorySlot 참조
• Slot.FuelAmount <= 0이면 early return (연료 아이템이 아닌 경우)
• FGameplayTag "Event.Interaction.AddFuel"을 RequestGameplayTag로 생성
• FGameplayEventData 구성:
  - EventTag = ActivationEventTag
  - Target = this (기차 자신)
  - Instigator = Interactor (플레이어)
  - EventMagnitude = FuelFromItem (아이템의 연료량)
• ASC->HandleGameplayEvent(Tag, &EventData)로 AddEnergy 어빌리티 이벤트 트리거
• Inventory->ClearSlot()으로 사용된 아이템 슬롯 비우기
핵심 패턴
GAS의 HandleGameplayEvent를 사용하여 이벤트 기반으로 어빌리티를 활성화합니다. AddEnergy 어빌리티는 WaitGameplayEvent 태스크로 "Event.Interaction.AddFuel" 태그를 대기하고 있다가, 이벤트 수신 시 EventMagnitude를 연료량으로 사용하여 Fuel 어트리뷰트를 증가시킵니다.
OnFuelChanged(const FOnAttributeChangeData& Data) ▶
구현 상세 (.cpp)
• OldValue, NewValue, Delta(New - Old) 산출
• TotalFuelGained += Delta (누적 — 음수 Delta도 합산)
• 서버(HasAuthority)에서 OldFuel >= 0 && NewFuel < 0 조건 성립 시:
  - GetWorld()->GetAuthGameMode<AAO_GameMode_Stage>()로 스테이지 게임모드 획득

AAO_newTrain (AO_Train과 별개의 기차 클래스)
UWorldSubsystem을 상속하며, 현재 월드(레벨)에 대해 자동으로 인스턴스화됩니다. GameInstance처럼 앱 전체가 아닌, 현재 레벨에서만 유효합니다. .cpp 파일에서 AO_newTrain.h를 직접 include하므로, AAO_Train이 아닌 AAO_newTrain과 연동됩니다.

RegisterTrain()은 null 체크 후 CachedTrain에 저장하고 UE_LOG로 등록 로그를 출력한 뒤 OnTrainRegistered를 Broadcast합니다. GetTrain()은 CachedTrain.Get()으로 TObjectPtr의 약한 참조를 반환하며, 기차가 소멸된 경우 nullptr을 반환합니다. 직접 FindActor를 사용하지 않아도 되므로 코드가 단순해지고, 등록 시점을 델리게이트로 정확히 알 수 있습니다.

이펙트 시스템
Niagara + Cascade VFX, SFX 지원. 딜레이 스폰과 액터 부착 모드 제공
UCLASS(Abstract)로 선언되어 직접 배치는 불가하고 반드시 상속받아 사용합니다. 생성자에서 bReplicates = true를 설정하고, MeshComponent(루트)와 InteractableMeshComponent(자식) 두 개의 StaticMeshComponent를 생성합니다. MeshComponent는 QueryAndPhysics 충돌, InteractableMeshComponent는 QueryOnly 충돌로 역할이 분리됩니다. AO_TraceChannel_Interaction 채널에 Block 응답을 설정하여 상호작용 트레이스에 감지됩니다.

상호작용 흐름: CanInteraction() → CanInteraction_BP() → 조건 통과 시 OnInteractionSuccess() 순서로 실행됩니다. CanInteraction_BP_Implementation은 부모의 CanInteraction, bInteractionEnabled 플래그, DisabledPlayers 배열을 순차적으로 확인합니다.

OnInteractionSuccess에서는 서버 Authority를 확인한 후 토글 모드에 따라 bIsActivated를 반전(토글)하거나 true로 고정합니다. StartInteractionAnimation으로 애니메이션을 시작하고, TriggerLinkedReactions로 연결된 AAO_PuzzleReactionActor들의 ActivateReaction/DeactivateReaction을 호출합니다. FAO_InteractionEffectSettings가 유효하면 MulticastPlayInteractionEffect RPC로 모든 클라이언트에 VFX/SFX를 동기화합니다.

이펙트 내부에서는 VFX/SFX 각각에 딜레이 타이머를 지원하며, WeakLambda 패턴으로 액터 소멸 시 안전하게 처리합니다. VFX는 Niagara(우선)와 Cascade를 모두 지원하며, bAttachVFXToActor 옵션으로 소켓 부착 또는 월드 스폰을 선택합니다.

AAO_PuzzleConditionChecker
부모: AActor (Blueprintable)
역할
GameplayTag 기반 퍼즐 조건을 검사하고 완료/실패를 판정
조건 유형
AND/OR 조합, 순서 조건 (ConditionOrderHistory), 시간 제한, 차단 태그
네트워크 설정
bReplicates=true, bAlwaysRelevant=true, 이동 복제 비활성화
완료 처리
IAbilitySystemInterface를 통해 TargetActors의 ASC에 CompletionTag를 LooseGameplayTag로 부여
레벨에 배치하는 범용 퍼즐 판정 액터입니다. 생성자에서 bReplicates=true, bAlwaysRelevant=true, SetReplicateMovement(false)로 설정되어 위치 변경 없이 상태만 복제합니다.

PuzzleElement(버튼/레버 등)가 OnPuzzleEvent(EventTag, Instigator)를 호출하면: 이미 완료되었거나 중복 태그면 무시하고, ActiveEvents에 태그를 추가합니다. bRequiresOrder인 조건에 대해서는 ConditionOrderHistory[조건인덱스]에 태그를 순서대로 기록합니다. TimeLimit > 0인 조건에 대해서는 StartConditionTimer로 타이머를 시작합니다.

CheckAllConditions()에서 각 조건을 CheckSingleCondition()으로 개별 검사합니다. bRequireAllConditions가 true이면 모든 조건이 만족(AND), false이면 하나라도 만족(OR)이면 CompletePuzzle()을 호출합니다. 조건이 만족되지 않으면 ShouldFailPuzzle()로 실패 조건을 검사합니다: AND 모드에서는 모든 필수 태그가 수신되었으나 순서/차단 태그로 인해 하나라도 불합격이면 실패, OR 모드에서는 모든 조건이 필수 태그를 전부 수신했으나 전부 불합격이면 실패입니다.

완료 시 CompletePuzzle()은 IAbilitySystemInterface를 통해 TargetActors의 ASC에 AddLooseGameplayTag(CompletionTag)를 부여하고, bDisableElementsOnComplete이면 IAO_Interface_PuzzleElement::SetInteractionEnabled(false)로 연결 요소들을 비활성화합니다.
FPuzzleCondition (USTRUCT, BlueprintType)

BlockingTags FGameplayTagContainer. ActiveEvents에 이 태그 중 하나라도 있으면(HasAny) CheckSingleCondition이 즉시 false를 반환합니다. 잘못된 버튼 조합 감지에 사용합니다.

CheckSingleCondition(Condition, ConditionIndex) — protected ▶
구현 상세 (.cpp)
1단계 — BlockingTags 체크:
• BlockingTags.Num() > 0이고 ActiveEvents.HasAny(BlockingTags)이면 즉시 false

2단계 — 순서 조건 체크 (bRequiresOrder && OrderedTags 존재 시):
• ConditionOrderHistory에 해당 인덱스가 없으면 false
• History 배열 크기가 OrderedTags 크기보다 작으면 false (아직 충분한 이벤트 미수신)
• History의 최근 N개(StartIndex = History.Num() - OrderedTags.Num())를 OrderedTags와 MatchesTagExact로 순서 비교
• 하나라도 불일치하면 false, 모두 일치하면 true 반환 (RequiredTags 검사 건너뜀)

3단계 — RequiredTags 체크 (순서 조건이 아닌 경우):
• RequiredTags.Num() > 0이면 ActiveEvents.HasAll(RequiredTags) 반환
• 아무 조건도 없으면(RequiredTags도 비어있으면) false 반환

순서 조건(bRequiresOrder)과 일반 조건(RequiredTags)은 상호 배타적으로 처리됩니다. 순서 조건이 활성화되면 RequiredTags는 검사하지 않습니다. 순서 비교 시 "최근 N개"만 확인하므로, 중간에 잘못된 태그가 들어와도 마지막 N개가 올바르면 통과할 수 있습니다.

GetCompletionProgress() const BlueprintPure ▶
구현 상세 (.cpp)
• Conditions.Num() == 0이면 0.0f 반환
• 모든 조건의 RequiredTags를 순회하며 TotalRequiredTags 합산
• 각 RequiredTag에 대해 ActiveEvents.HasTag() 체크 → 매칭 시 AcquiredTags 증가
• TotalRequiredTags == 0이면 0.0f 반환
• static_cast<float>(AcquiredTags) / static_cast<float>(TotalRequiredTags) 반환 (0.0 ~ 1.0)
제한사항
순서 조건(OrderedTags)의 진행도는 이 계산에 포함되지 않습니다. RequiredTags 기반으로만 진행도를 산출합니다.
