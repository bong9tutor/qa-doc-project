FAO_STCond_CanAmbush
부모: FStateTreeAIConditionBase
역할
Stalker가 기습 공격 가능한 위치(플레이어 등 뒤/옆)에 있는지 판단
사용 AI
Stalker
판단 방식
플레이어 전방 벡터와 AI 방향의 내적(Dot Product) 비교
DisplayName
AO Can Ambush
플레이어에서 Stalker로의 방향 벡터와 플레이어의 전방 벡터를 내적하여 등 뒤(BackAngleCosine, 기본 -0.3 = 약 110도) 또는 옆(SideAngleCosine, 기본 0.0 = 90도)에 있는지 판단합니다. ChaseTarget이 없으면 시야 내 가장 가까운 플레이어를 대상으로 사용합니다. bInvert로 결과를 반전할 수 있습니다.
심층 분석: InstanceData 구조체가 분리된 패턴(FAO_STCond_CanAmbush_InstanceData)을 사용합니다. using FInstanceDataType으로 별칭을 선언하고, GetInstanceDataType()에서 StaticStruct()를 반환하여 StateTree 프레임워크가 인스턴스 데이터를 자동 관리합니다. TestCondition에서 Ctrl이나 Target이 null일 때 InstanceData.bInvert를 반환하므로, bInvert=false면 안전하게 false, bInvert=true면 "기습 불가능"을 "기습 가능"으로 반전하여 반환합니다. GetAggressiveController()는 Owner가 Controller인지 Pawn인지에 따라 이중 캐스트 경로를 제공하는 공통 패턴입니다.

수학적 분석: Dot ≤ BackAngleCosine(-0.3)이면 등 뒤, Dot ≤ SideAngleCosine(0.0) && Dot > BackAngleCosine이면 옆입니다. 두 조건이 OR로 결합되므로, 결국 Dot ≤ SideAngleCosine(0.0)이면 true가 됩니다. 즉 BackAngleCosine 조건은 사실상 중복이며, SideAngleCosine만으로 충분합니다. 이는 의도적으로 세분화하여 향후 등 뒤/옆을 다르게 처리할 수 있도록 한 설계로 보입니다.

플레이어의 전방 벡터와 AI 방향의 Dot Product를 계산하여 기습 가능 여부를 반환합니다.
상세 흐름
(1) Context.GetInstanceData<FAO_STCond_CanAmbush_InstanceData>(*this)로 인스턴스 데이터 획득
(2) GetAggressiveController(Context)로 Ctrl 획득 — null이면 bInvert 반환
(3) Ctrl->GetChaseTarget()으로 Target 시도, null이면 Ctrl->GetNearestPlayerInSight()로 대체
(4) StalkerPawn = Ctrl->GetPawn() — Target 또는 StalkerPawn null이면 bInvert 반환
(5) DirToStalker = (StalkerPawn위치 - Target위치).GetSafeNormal()
(6) Dot = DotProduct(Target->GetActorForwardVector(), DirToStalker)
(7) bCanAmbush = (Dot ≤ BackAngleCosine) || (Dot ≤ SideAngleCosine && Dot > BackAngleCosine)
(8) bInvert ? !bCanAmbush : bCanAmbush
null 안전성
3단계 null 체크: Ctrl, Target(두 번째 시도 포함), StalkerPawn. 모두 실패 시 bInvert 반환으로 안전 종료.

(2) Cast<AAO_AggressiveAICtrl>(Owner) 시도 — 성공 시 직접 반환
(3) Cast<APawn>(Owner) → Pawn->GetController() → Cast<AAO_AggressiveAICtrl>
(4) 모두 실패 시 nullptr 반환
패턴 분석
이 이중 캐스트 패턴(Controller 직접 / Pawn의 Controller)은 STCond_InAttackRange, STCond_PlayerTooClose, STCond_PlayerInSight에서도 동일하게 사용됩니다. StateTree Owner가 Controller일 수도 Pawn일 수도 있기 때문입니다.

AggressiveCtx의 모든 전투 데이터를 포함하면서, Stalker 전용 데이터를 추가합니다: 엄폐 위치, 도주 위치, 플레이어 시선 감지(Hysteresis 적용), 도주 상태.
심층 분석: 유일하게 상속 관계를 사용하는 Evaluator입니다. 인스턴스 데이터도 상속합니다: FAO_STEval_StalkerCtx_InstanceData : public FAO_STEval_AggressiveCtx_InstanceData. 이로 인해 AggressiveCtx의 모든 출력 변수(bIsChasing, CurrentTarget 등)를 그대로 사용하면서 Stalker 전용 변수를 추가합니다.

Hysteresis 시선 감지: StalkerCtrl->UpdateLookingPlayerWithHysteresis(DeltaTime)를 호출합니다. DeltaTime을 받아 시간 기반 안정화를 적용하여, 플레이어가 잠깐 시선을 돌렸다가 다시 보는 경우 빈번한 상태 변경을 방지합니다.

다중 플레이어 엄폐: GetAllLookingPlayers()로 나를 보는 모든 플레이어를 수집한 후, FindHideLocationFromMultiple(1500, LookingPlayers)로 모든 시선에서 벗어나는 위치를 계산합니다. 아무도 보지 않으면 CurrentTarget.Get()을 사용하여 단일 대상 기반 엄폐 위치를 계산합니다. 1500은 하드코딩된 엄폐 검색 반경입니다.

도주 위치: 도주 중이 아닐 때 RetreatLocation = Stalker->GetActorLocation()으로 현재 위치를 설정합니다. 이는 도주가 시작되었을 때 FindRetreatLocation()이 호출될 때까지의 임시값입니다.

11개 출력: 가장 많은 출력 변수를 가진 Evaluator입니다. bShouldPrioritizeWeapon과 NearestWeaponLocation은 Controller에서 직접 가져오며, 무기 줍기 우선순위 결정은 Controller의 로직에 위임됩니다.
