Reliable vs Unreliable 가이드

유형 언제 사용 예시
Server, Reliable 게임플레이 상태 변경 (반드시 서버에 도달해야 함) ServerMeleeAttack, ServerTriggerInteract
Multicast, Reliable 모든 클라이언트가 반드시 받아야 하는 핵심 이벤트 MulticastPlayInteractionMontage, MulticastOnDeath
Multicast, Unreliable 코즈메틱/시각 효과 (패킷 손실 허용) MulticastPlayMeleeAnim, MulticastPlayHitSound
Client, Reliable 서버가 특정 클라이언트에게만 지시 ClientUpdateHP, Client_PrepareForTravel
Reliable RPC는 재전송 보장이지만 대역폭을 더 소비합니다. 매 프레임 호출하는 애니메이션/이펙트에 Reliable을 쓰면 대역폭 포화로 연결이 끊길 수 있습니다.

Replicated 변수 패턴

GetLifetimeReplicatedProps() 패턴
// AO_PlayerState.cpp - 복제 변수 등록
void AAO_PlayerState::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);

    // DOREPLIFETIME: 모든 클라이언트에 항상 복제
    DOREPLIFETIME(AAO_PlayerState, bLobbyIsReady);
    DOREPLIFETIME(AAO_PlayerState, bIsAlive);
    DOREPLIFETIME(AAO_PlayerState, DeathCount);
    DOREPLIFETIME(AAO_PlayerState, PersistentInventory);
    DOREPLIFETIME(AAO_PlayerState, CharacterCustomizingData);
}
DOREPLIFETIME vs DOREPLIFETIME_CONDITION
// DOREPLIFETIME - 무조건 모든 클라이언트에 복제
DOREPLIFETIME(AAO_GameState, SharedReviveCount);
DOREPLIFETIME(AAO_GameState, bIsStageFailed);

// DOREPLIFETIME_CONDITION - 조건부 복제 (대역폭 절약)
// COND_OwnerOnly: 소유 클라이언트에만 복제
DOREPLIFETIME_CONDITION(AAO_PlayerState,
    PersistentHealth, COND_OwnerOnly);

// COND_InitialOnly: 최초 1회만 복제
DOREPLIFETIME_CONDITION(AAO_PlayerState,
    bIsLobbyHost, COND_InitialOnly);
현재 프로젝트에서는 대부분 DOREPLIFETIME (무조건 복제)을 사용합니다. 최적화가 필요한 경우 COND_OwnerOnly, COND_InitialOnly 등을 고려하세요.
OnRep 콜백 패턴
// 1. UPROPERTY에 ReplicatedUsing 지정
UPROPERTY(ReplicatedUsing=OnRep_LobbyIsReady)
bool bLobbyIsReady;

// 2. OnRep 콜백 구현 (클라이언트에서 값 변경 시 자동 호출)
void AAO_PlayerState::OnRep_LobbyIsReady()
{
    OnLobbyReadyChanged.Broadcast(bLobbyIsReady);
    RefreshLobbyReadyBoard();
}

// 주의: 서버(Authority)에서는 OnRep이 자동 호출되지 않음!
// 서버에서도 같은 로직이 필요하면 수동 호출 필수:
void AAO_GameState::SetStageFailed()
{
    bIsStageFailed = true;
    OnRep_IsStageFailed();  // 서버 수동 호출
}
서버에서 OnRep이 자동 호출되지 않는 것은 흔한 버그 원인입니다. 이 프로젝트에서도 SetStageFailed, SetGameClear, AddTeamDeathCount 등에서 수동 호출 패턴을 사용합니다.
Replicated 배열 (PersistentInventory 예시)
// AO_PlayerState.h - 배열 복제 선언
UPROPERTY(Replicated)
TArray<FInventorySlot> PersistentInventory;

// AO_PlayerState.cpp - 복제 등록
DOREPLIFETIME(AAO_PlayerState, PersistentInventory);

// 레벨 이동 전 인벤토리 저장
void AAO_PlayerState::SaveInventoryBeforeTravel(
    UAO_InventoryComponent* Inv)
{
    PersistentInventory = Inv->Slots;
}

// CopyProperties로 새 PlayerState에 이관
void AAO_PlayerState::CopyProperties(APlayerState* PlayerState)
{
    Super::CopyProperties(PlayerState);
    if (bInventoryShouldPersist)
        PS->PersistentInventory = this->PersistentInventory;
}
TArray는 전체 배열이 복제됩니다. 요소가 많으면 대역폭 비용이 큽니다. FFastArraySerializer 사용으로 델타 복제가 가능합니다.

동시 접근 충돌 해결

문제: 두 플레이어가 동시에 같은 아이템을 줍는다면?
Player A와 Player B가 동시에 E키를 눌러 같은 연료통에 ServerInteract RPC를 보냅니다.
네트워크 지연으로 두 RPC가 거의 동시에 서버에 도착할 수 있습니다. 
// 서버 권한(Authority)이 충돌을 해결합니다
// 먼저 도착한 RPC가 처리되고, 두 번째는 무시됩니다

// InteractionComponent - 서버에서 유효성 검사
void UAO_InteractionComponent::ServerTriggerInteract_Implementation(
    AActor* TargetActor)
{
    // 1. null/유효하지 않은 타겟 차단
    if (!TargetActor || !TargetActor->IsValidLowLevel())
    {
        return;  // 이미 수집된 아이템은 Destroy됨
    }

    // 2. 어빌리티 시스템으로 실행 위임
    bIsHoldingInteract = true;
    ASC->HandleGameplayEvent(ExecuteTag, &Payload);
}

// FuelActor - bCollected 플래그로 중복 방지
void AAO_FuelActor::Interact(ACharacter* Interactor)
{
    if (!bCollected)  // 서버 단일 스레드 → 경합 없음
    {
        bCollected = true;
        FuelManager->AddFuel(FuelValue);
        MulticastOnCollected();
    }
    // 두 번째 호출 시 bCollected==true → 무시
}
서버 권한 패턴 요약
원칙 서버에 먼저 도착한 RPC가 승리합니다 (First-Come, First-Served).
가드 서버에서 IsValid / bCollected / HasAuthority 체크 후 처리합니다.
안전 UE5 게임 스레드는 싱글 스레드이므로 두 RPC가 진짜 "동시"에 실행되지 않습니다.
