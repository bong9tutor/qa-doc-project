FAO_STTask_AIMoveTo
부모: FStateTreeAIActionTaskBase
역할
지정된 위치로 AI를 NavMesh 기반 이동시키는 범용 Task
DisplayName
AO AI Move To
인스턴스 데이터
FAO_STTask_AIMoveTo_InstanceData
실행 위치
서버 (AIController 필요)
TargetLocation과 AcceptanceRadius를 인스턴스 데이터로 받아 AIController::MoveToLocation()을 호출합니다. EnterState에서 이동을 시작하고, Tick에서 PathFollowingComponent의 상태를 확인하여 도착 시 Succeeded를 반환합니다. ExitState에서는 이동 중이었다면 StopMovement()로 정리합니다.
심층 분석: GetAIController()는 Context.GetOwner()가 Pawn인지 Controller인지 양쪽 모두 대응합니다. AIMovementBase와 동일한 패턴이지만 상속 관계가 아닌 독립 구현입니다. Tick에서 PathComp가 Idle이면 Succeeded를 반환하는데, 경로 계산 실패로 Idle이 되는 경우도 Succeeded로 처리되어 의도치 않은 성공 반환 가능성이 있습니다. TargetLocation.IsZero() 체크는 원점이 유효한 목표인 경우를 배제합니다.

bIsMovingboolUPROPERTY현재 이동 중 여부. ExitState에서 StopMovement 호출 판단에 사용. EnterState에서만 true로 설정됨

흐름

심층
ensureMsgf는 개발 빌드에서만 assert하고 Shipping에서는 로그 없이 false 반환. AlreadyAtGoal 즉시 Succeeded는 StateTree 상태 전이를 EnterState 한 프레임에서 완료시킵니다.
Tick()Override▶
흐름
GetAIController → GetPathFollowingComponent (null 시 Failed) → GetStatus() == Idle이면 bIsMoving=false → Succeeded, 아니면 Running
심층
PathComp가 Moving/Paused/Waiting 어느 상태든 Running을 반환합니다. Idle은 "이동 완료" 또는 "요청 없음" 모두 해당하므로, 외부에서 StopMovement()를 호출한 경우에도 Succeeded가 반환됩니다.

FAO_STTask_AIRoam
부모: FStateTreeAIActionTaskBase
역할
NavMesh 내 랜덤 위치로 이동 → 대기 → 재이동을 반복하는 배회 Task
DisplayName
AO AI Roam
종별 분기
Crab이면 주변 아이템 감지 시 Failed 반환, Stalker 분기는 주석 처리됨
실행 패턴
EnterState에서 이동 시작 → Tick에서 이동/대기 상태 전환 → Running 유지
GetRandomReachablePointInRadius()로 NavMesh 내 유효 위치를 찾아 이동합니다. 목적지 도착 후 WaitTimeAtDestination만큼 대기한 뒤 새로운 배회 위치로 이동합니다. Running을 계속 반환하여 StateTree가 다른 조건으로 전환할 때까지 배회를 유지합니다.
심층 분석: Crab 분기에서 FindNearbyItem(1500.f) 하드코딩이 있습니다 — CrabPickup의 SearchRadius(1500)와 일치하지만 연동 없이 매직 넘버입니다. Stalker 분기(else if)에는 주석만 있고 실제 로직이 없어, Stalker가 AIRoam을 사용하면 천장 전환 없이 바닥만 배회합니다 (Stalker_Roam이 별도로 존재하므로 정상). 대기 중 새 위치를 찾지 못해도(IsZero) 이동을 시작하지 않고 대기를 연장하여 무한 대기 가능성이 낮지만, StartMovingToLocation이 false를 반환하면 bIsWaiting이 false, bIsMoving도 false인 교착 상태에 빠집니다.

역할

DisplayName
AO AI Chase
완료 조건
공격 범위 도달 시 Succeeded, 타겟 없음/기절 시 Failed

AggressiveAICtrl에서 ChaseTarget을 가져와 FAIMoveRequest로 추격합니다. Tick마다 PathUpdateInterval 간격으로 경로를 갱신하고, 가장 가까운 플레이어로 타겟을 업데이트합니다.
심층 분석: StartChasing()은 GoalLocation 기반 FAIMoveRequest를 사용합니다 (GoalActor가 아님). 따라서 경로 갱신 시점 사이에 플레이어가 빠르게 이동하면 구 위치를 추격합니다. 0.3초 갱신 간격은 이를 완화하지만 MoveToActor 대비 정확도가 떨어집니다. UpdateChaseTargetToNearest()가 경로 갱신 후에 호출되므로, 새 타겟으로 전환되어도 다음 갱신 주기까지 구 타겟 방향으로 이동합니다. EnterState에서 StartChasing 실패 시에도 Running을 반환하여 이동 없이 Tick이 계속 실행됩니다. ExitState에서 Controller::StopMovement()가 AggressiveAICtrl의 멤버 함수로 호출됩니다.

bIsMovingboolUPROPERTY이동 중 여부. ExitState 정리에 사용

심층
StartChasing 실패 시에도 Running을 반환하는데, Tick에서 공격 범위 체크나 타겟 갱신이 계속 진행되므로 다음 갱신 주기에 재시도됩니다.

겹침 방지
bPreventOverlap: 전용 UAudioComponent 재사용 vs SpawnSound AutoDestroy
EnterState에서 AudioComponent 생성(겹침 방지 모드) + bPlayOnEnter이면 Timer=0. Tick에서 Timer 감소 → 0 이하 시 재생.
심층 분석: bPreventOverlap=true 모드에서 EnterState에서 NewObject<UAudioComponent>를 생성하고 RegisterComponent합니다. ExitState에서 bStopOnExit에 따라 Stop+DestroyComponent 또는 DestroyComponent만 수행합니다. bStopOnExit=false여도 컴포넌트는 파괴하여 댕글링을 방지합니다. ComputeNextInterval에서 ClampMin 0.1초를 보장합니다. SpawnSoundAttached/SpawnSoundAtLocation의 UE 5.6 시그니처가 KSJ 주석으로 확인되어 있습니다. 타이머가 카운트다운 방식이므로 bPlayOnEnter=true 시 Timer=0으로 설정하면 첫 Tick에서 즉시 재생됩니다.
