서버 권한 모델 (Authority Model)
HasAuthority() 패턴
UE5 Listen Server에서 게임플레이 상태 변경은 반드시 서버(Authority)에서만 수행해야 합니다. HasAuthority()는 현재 액터가 서버 인스턴스인지 확인하는 가드 함수로, 상태 변경 함수 진입부에서 반드시 체크합니다. 
// AO_GameState.cpp — 서버 전용 가드 패턴
void AAO_GameState::SetSharedReviveCount(int32 InValue)
{
    if (HasAuthority() == false)
    {
        return;  // 클라이언트에서는 즉시 반환
    }
    SharedReviveCount = NewValue;
    // ... 서버에서만 값 변경 후 Replication 으로 동기화
}

// AO_PlayerState.cpp — 사망 처리도 서버 전용
void AAO_PlayerState::SetIsAlive(bool bInIsAlive)
{
    if (HasAuthority())
    {
        bIsAlive = bInIsAlive;
        OnRep_IsAlive();  // 서버는 OnRep 자동호출 안됨 → 수동
    }
}
Network Role 설명 HasAuthority()
ROLE_Authority 서버에 존재하는 정본(Authoritative) 인스턴스 true
ROLE_AutonomousProxy 소유 클라이언트의 로컬 복제본 (입력 예측 가능) false
ROLE_SimulatedProxy 다른 클라이언트가 보는 원격 복제본 (보간 처리) false
클래스 존재 위치 비고
AO_GameMode_* 서버 전용 클라이언트에 존재하지 않음. 게임 규칙/레벨 전환 담당
AO_GameState 서버 + 모든 클라이언트 공유 상태 (부활칩, 스테이지 성공/실패). 서버가 값 변경, 클라에 복제
AO_PlayerState 서버 + 모든 클라이언트 플레이어별 상태 (레디, 생존, 데스카운트). 모든 플레이어에게 복제
AO_PlayerController_* 서버 + 소유 클라이언트 Client RPC 수신 대상. 다른 클라이언트에는 존재하지 않음
AO_PlayerCharacter 서버 + 모든 클라이언트 소유 클라: AutonomousProxy, 타 클라: SimulatedProxy
AO_AISubsystem 서버 전용 AI 로직(StateTree, EQS)은 서버에서만 실행
서버 권한 원칙: 게임플레이 상태(체력, 부활칩, 스테이지 클리어 등)는 반드시 서버에서 변경하고, Replication이나 Multicast RPC로 클라이언트에 전파합니다. 클라이언트가 직접 상태를 변경하면 치팅이 가능해지고, 클라이언트 간 불일치가 발생합니다.
Replication 조건 (DOREPLIFETIME_CONDITION)
GetLifetimeReplicatedProps() 패턴
Replicated 프로퍼티는 GetLifetimeReplicatedProps()에서 등록해야 엔진이 네트워크로 동기화합니다. 매크로를 통해 누구에게, 언제 복제할지 조건을 지정할 수 있습니다.
매크로 / 조건 동작 사용 시점
COND_None 모든 클라이언트에 항상 복제 공유 게임 상태 (체력, 스테이지 결과)
COND_OwnerOnly 소유자 클라이언트에만 복제 인벤토리, 개인 설정 등 본인만 필요한 데이터
COND_SkipOwner 소유자를 제외한 나머지에 복제 소유자는 로컬 예측으로 처리, 타 클라만 동기화 필요 시
COND_InitialOnly 최초 1회만 복제 (이후 변경 무시) 초기 설정값, 변하지 않는 속성 (팀 ID 등)
REPNOTIFY_Always 값이 같아도 OnRep 호출 보장 GAS AttributeSet (체력이 0→0일 때도 UI 갱신 필요)
AO_GameState — DOREPLIFETIME (COND_None 기본)
void AAO_GameState::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);

    DOREPLIFETIME(AAO_GameState, SharedReviveCount);
    DOREPLIFETIME(AAO_GameState, bIsStageFailed);
    DOREPLIFETIME(AAO_GameState, RunResetTrigger);
    DOREPLIFETIME(AAO_GameState, CurrentFindHintNum);
    DOREPLIFETIME(AAO_GameState, bHint1);
    DOREPLIFETIME(AAO_GameState, bHint2);
    DOREPLIFETIME(AAO_GameState, bHint3);
}
AO_PlayerState — DOREPLIFETIME (COND_None 기본)
void AAO_PlayerState::GetLifetimeReplicatedProps(
    TArray<FLifetimeProperty>& OutLifetimeProps) const
{
    Super::GetLifetimeReplicatedProps(OutLifetimeProps);

    DOREPLIFETIME(AAO_PlayerState, bLobbyIsReady);
    DOREPLIFETIME(AAO_PlayerState, LobbyJoinOrder);
    DOREPLIFETIME(AAO_PlayerState, bIsLobbyHost);
    DOREPLIFETIME(AAO_PlayerState, bIsAlive);
    DOREPLIFETIME(AAO_PlayerState, CharacterCustomizingData);
    DOREPLIFETIME(AAO_PlayerState, PersistentInventory);
    DOREPLIFETIME(AAO_PlayerState, DeathCount);
    DOREPLIFETIME(AAO_PlayerState, bHasPersistentHealth);
    DOREPLIFETIME(AAO_PlayerState, PersistentHealth);
}
주의: DOREPLIFETIME에 조건 없이 등록하면 기본값 COND_None이 적용되어 모든 클라이언트에 복제됩니다. 대역폭 최적화가 필요하면 COND_OwnerOnly 등 조건을 명시하세요. 또한 UPROPERTY(Replicated) 또는 UPROPERTY(ReplicatedUsing=OnRep_*) 선언과 반드시 쌍으로 사용해야 합니다.
Reliable vs Unreliable RPC
신뢰성 선택 기준
RPC의 Reliable/Unreliable 선택은 게임 정확성과 대역폭 사이의 트레이드오프입니다. 잘못된 선택은 패킷 유실 시 게임 상태 불일치 또는 대역폭 포화로 이어집니다.
구분 Reliable Unreliable
전송 보장 패킷 유실 시 재전송. 순서 보장 유실되면 무시. 순서 보장 안 됨
대역폭 ACK 오버헤드 + 재전송 큐 소비 최소 오버헤드, 한 번 전송 후 잊음
사용 대상 상태 변경, 게임 흐름 (승리, 상호작용, 데미지) 시각/청각 효과 (애니메이션, VFX, 사운드)
포화 위험 Reliable 큐 가득 차면 연결 끊김 프레임 하나 빠질 뿐 문제 없음
Unreliable 예시 — MulticastPlayMeleeAnim (코스메틱)
// 애니메이션은 유실되어도 게임 상태에 영향 없음 → Unreliable
UFUNCTION(NetMulticast, Unreliable)
void MulticastPlayMeleeAnim();

// 구현: 모든 클라이언트에서 몽타주 재생
void AAO_PlayerCharacter::MulticastPlayMeleeAnim_Implementation()
{
    if (MeleeAnimMontage)
        GetMesh()->GetAnimInstance()->Montage_Play(MeleeAnimMontage);
}
Reliable 예시 — MulticastOnVictory (게임 흐름)
// 승리 결과는 모든 클라에게 반드시 전달되어야 함 → Reliable
UFUNCTION(NetMulticast, Reliable)
void MulticastOnVictory();

// 구현: 모든 클라이언트에서 승리 UI 표시
void AAO_GameMode_InGame::MulticastOnVictory_Implementation()
{
    ShowVictoryUI();
}
Reliable 예시 — ServerRPC_SetInputState (서버 상태 변경)
// 입력 상태를 서버에 전달 — 이동 판정에 영향 → Reliable
UFUNCTION(Server, Reliable)
void ServerRPC_SetInputState(bool bWantsToSprint, bool bWantsToWalk);
판단 기준: "이 RPC가 유실되면 게임 결과가 달라지는가?" — Yes면 Reliable, No면 Unreliable. Reliable RPC를 매 틱 호출하면 큐 포화로 연결이 끊길 수 있으므로, 반복적 코스메틱 호출에는 절대 사용하지 마세요.

4단계: 서버 측 값 변경 + 사망 판정
// PostGameplayEffectExecute — 서버에서만 실행
void UAO_PlayerCharacter_AttributeSet::PostGameplayEffectExecute(
    const FGameplayEffectModCallbackData& Data)
{
    if (Data.EvaluatedData.Attribute == GetHealthAttribute())
    {
        const float NewHealth = GetHealth();
        if (NewHealth <= 0.f)
        {
            if (Owner->HasAuthority()) // 서버 전용
            {
                OnPlayerDeath.Broadcast();
            }
        }
        SetHealth(FMath::Clamp(NewHealth, 0.f, GetMaxHealth()));
    }
}
Ability 활성화 흐름 요약
Client
입력 감지 → TryActivateAbilityByTag() 호출
Server RPC
서버에서 Ability 활성화 → GameplayEffect 적용 → Attribute 변경
Replication
변경된 Attribute가 모든 클라이언트로 복제 → OnRep_*() 호출
Multicast
(선택) 코스메틱 효과(애니메이션, VFX)는 Unreliable Multicast로 별도 전파
UI Update
OnRep 콜백 → GAMEPLAYATTRIBUTE_REPNOTIFY → ASC 내부 델리게이트 → HUD 위젯 갱신
GAS 복제 핵심: DOREPLIFETIME_CONDITION_NOTIFY에서 REPNOTIFY_Always를 사용하는 이유는, GAS 내부 예측 시스템이 서버 값과 클라이언트 예측값을 비교/보정하기 위해 값이 동일해도 OnRep 호출이 필요하기 때문입니다. 일반 DOREPLIFETIME은 값이 변하지 않으면 OnRep을 호출하지 않습니다.
