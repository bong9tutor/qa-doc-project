AAO_CustomizingCharacter
부모: AActor
역할
캐릭터 커스터마이징 프리뷰 전용 액터. 로비/메뉴에서 외형 미리보기를 담당
실행 위치
로컬 클라이언트 전용 (bReplicates = false, PrimaryActorTick.bCanEverTick = false)
특징
Tick/Replicate 비활성. Mutable CustomizableObject 기반. USceneComponent 루트
카메라
SpringArm(TargetArmLength=200cm) + Camera(SocketName 부착)으로 프리뷰 시점 제공
AActor를 직접 상속하는 커스터마이징 전용 액터입니다. ACharacter가 아닌 AActor를 상속하여 이동/충돌/물리 관련 오버헤드 없이 순수 비주얼 프리뷰만 수행합니다. 게임플레이에 참여하지 않으며, 로비/메인메뉴에서 캐릭터 외형을 미리보기하는 데 사용합니다. DefaultSkeletalMesh를 루트 하위에 배치하고, BaseSkeletalMesh, BodySkeletalMesh, HeadSkeletalMesh가 계층적으로 부착됩니다. Mutable의 UCustomizableSkeletalComponent(Body/Head)가 실시간 외형 변경을 처리하며, DummyCustomComponent를 통해 커스터마이징 데이터를 외부에 노출합니다.
심층 분석 -- 생성자 구성 순서
생성자에서 컴포넌트 생성 순서: Root(USceneComponent) → SpringArm → Camera → DefaultSkeletalMesh → BaseSkeletalMesh → BodySkeletalMesh + BodyComponent → HeadSkeletalMesh + HeadComponent → DummyCustomComponent. Root가 SetRootComponent로 루트 지정되며, 모든 스켈레탈 메시는 DefaultSkeletalMesh 아래에 계층적으로 부착됩니다. BodyComponent와 HeadComponent는 각각 SetComponentName("Body"/"Head")으로 Mutable 시스템에서 파트를 식별합니다.
QA 검토 포인트 -- 복제 불일치
bReplicates = false인데 BodyComponent, HeadComponent, DummyCustomComponent에 SetIsReplicated(true)가 설정되어 있습니다. 액터 자체가 복제되지 않으면 하위 컴포넌트의 복제도 무의미합니다. 원래 주석 처리된 BaseSkeletalMesh/BodySkeletalMesh/HeadSkeletalMesh의 SetIsReplicated는 의도적으로 제거한 것으로 보이나, CustomizableSkeletalComponent와 DummyCustomComponent의 복제 설정은 정리되지 않은 상태입니다. 기능에는 영향이 없으나 코드 의도가 불명확합니다.

지원 캐릭터
ECharacterMesh: Elsa(기본), Anka, Bruce, Cameron (4종)
UActorComponent를 상속하며, PlayerCharacter에 부착되어 캐릭터 외형(메시, 헤어, 의상)을 런타임에 변경합니다. Mutable의 UCustomizableObjectInstance를 캐릭터 종류별로 생성하여 관리하고, FCustomizingData 구조체로 현재 외형 상태를 추적합니다. 심층 분석 -- 네트워크 동기화 흐름
1. 클라이언트 UI가 ServerRPC_ChangeCustomizing을 호출
2. 서버에서 CustomizingData를 갱신하고 ApplyCustomizingData() 실행 (서버 측 즉시 적용)
3. DOREPLIFETIME으로 CustomizingData가 다른 클라이언트에 복제
4. 클라이언트에서 OnRep_CustomizingData → ApplyCustomizingData() 실행
주의: ServerRPC에서 직접 Apply를 호출하는 이유는 Listen Server에서 서버 자신도 외형을 볼 수 있어야 하기 때문입니다. OnRep은 서버에서는 호출되지 않으므로, 서버 측 적용을 위해 RPC 구현부에서 직접 Apply합니다.

UAO_AbilityTask_WaitForInvalidInteraction
부모: UAbilityTask
역할
홀딩 상호작용 중 플레이어의 이탈(방향/위치 변화)을 감지하는 태스크
검증 주기
0.1초마다 PerformCheck 실행 (WeakLambda 타이머)
검증 기준
시선 각도(AcceptanceAngle), XY 거리(DistSquared2D), Z 거리(Abs + CrouchedHalfHeight) 3가지
결과
범위 초과 시 경고 로그 + OnInvalidInteraction.Broadcast + EndTask
UAbilityTask를 상속하며, 홀딩 상호작용(예: 4초간 F키 누르기) 중에 플레이어가 뒤돌아보거나 멀리 이동하면 상호작용을 취소합니다. Activate() 시점에 캐릭터의 전방 벡터(2D)와 위치를 스냅샷하고, 0.1초마다 현재 상태와 비교합니다.
심층 분석 -- 3가지 유효성 검증
1. 각도 검증 (bValidAngle2D): CalculateAngle2D() ≤ AcceptanceAngle. 시작 시 캐릭터 전방 벡터와 현재 전방 벡터의 내적을 DegAcos로 각도 변환. 플레이어가 뒤돌아보면 즉시 취소.
2. XY 거리 검증 (bValidDistanceXY): DistSquared2D ≤ AcceptanceDistance². 제곱 비교로 sqrt 비용 회피. 플레이어가 옆으로 이동하면 취소.
3. Z 거리 검증 (bValidDistanceZ): |ΔZ| ≤ AcceptanceDistance + CrouchedHalfHeight. 앉기 동작은 Z축 변화를 유발하므로 앉기 높이만큼 여유를 줍니다. 점프나 낙하 시 취소.
세 조건 모두 통과해야 홀딩이 유지됩니다.

UAO_GameplayAbility_Outline
부모: UGameplayAbility
역할
특정 타입의 액터에 일시적 외곽선 효과를 적용하는 어빌리티
실행 정책
LocalPredicted + InstancedPerActor
시각 효과
PostProcess 머티리얼 2개(기본+Clip) + CustomDepth StencilValue(240) 기반 외곽선
지속 시간
AbilityDuration(기본 3초) 후 WaitDelay Task → OnDurationFinished → EndAbility
UGameplayAbility를 상속하며, 활성화 시 로컬 클라이언트에서 PostProcess 머티리얼을 캐릭터의 PostProcessComponent에 추가하고, OutlineActorClass에 해당하는 모든 월드 액터에 CustomDepth를 설정합니다. 기본 색상은 어두운 적색(0.05, 0, 0.02)이고 외곽선은 밝은 적색(1, 0, 0.06)입니다.
심층 분석 -- 2개 PostProcess 머티리얼 구조
OutlineMaterial: 기본 외곽선 렌더링. CustomDepth 오브젝트에 반투명 색상(Opacity=0.2) + 외곽선(OutlineOpacity=0.5) 적용
OutlineMaterial_Clip: 벽 뒤에 가려진 오브젝트의 외곽선 렌더링 (클리핑 처리). 같은 파라미터 공유
두 머티리얼 모두 DynamicMaterialInstance로 생성되어 런타임에 파라미터 조절이 가능합니다. WeightedBlendable(Weight=1.0)로 PPC에 추가되어 다른 포스트프로세스 효과와 합성됩니다.

StencilValue 체계: 240 vs 250

TActorIterator 성능 고려사항
ActivateAbility에서 TActorIterator로 OutlineActorClass의 모든 월드 인스턴스를 순회합니다. 이는 어빌리티 활성화 시 1회만 실행되므로 일반적으로 문제없으나, OutlineActorClass가 AActor처럼 광범위한 클래스로 설정되면 모든 액터를 순회하여 프레임 드롭이 발생할 수 있습니다. 일반적으로는 특정 퍼즐 오브젝트 클래스 등 좁은 범위로 설정합니다.
WaitDelay Task 활용
엔진 내장 UAbilityTask_WaitDelay를 사용하여 시간 경과를 대기합니다. 이 태스크는 GAS 프레임워크 내에서 관리되므로 어빌리티가 취소되면 타이머도 자동 정리됩니다. SetTimer를 직접 사용하는 것보다 GAS와의 통합이 깔끔합니다. WaitDelay는 서버/클라이언트 양쪽에서 실행되어, InstancedPerActor + LocalPredicted 정책과 함께 양쪽에서 정확한 타이밍에 EndAbility가 호출됩니다.
