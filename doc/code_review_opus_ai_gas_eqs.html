UAO_GA_AIAttackBase
부모: UGameplayAbility
역할
모든 AI 적의 공통 근접 공격 어빌리티 기반 클래스
GAS 카테고리
InstancedPerActor · ServerInitiated
실행 위치
ServerInitiated — 서버에서 시작하여 실행 및 히트 판정 수행
주요 책임
공격 설정(FEnemyAttackConfig) 로드, 몽타주 재생, SphereTrace 히트 판정, 데미지/넉백 적용, HitReact 이벤트 전송
AAO_AICharacterBase의 GetCurrentAttackConfig()를 통해 공격 데이터(데미지, 몽타주 등)를 가져옵니다. Event.Combat.Confirm 이벤트를 WaitGameplayEvent 태스크로 대기하고, AnimNotify 타이밍에 맞춰 SphereTrace를 수행합니다. 무적(Status.Invulnerable) 확인 후 데미지를 적용하고, 넉백 강도에 따라 Light 또는 Knockdown HitReact 이벤트를 전송합니다. 타격음은 MulticastPlayHitSound(NetMulticast, Unreliable)로 모든 클라이언트에서 재생합니다. Werewolf_Attack 등의 자식 클래스가 OnTargetHit/ApplyDamageAndKnockback을 오버라이드하여 동작을 확장합니다.
[심층 분석] GAS 파이프라인 상세: 생성자에서 InstancingPolicy=InstancedPerActor와 NetExecutionPolicy=ServerInitiated를 설정합니다. 이는 AI당 하나의 인스턴스가 서버에서 생성되어 실행됨을 의미합니다. ActivateAbility 흐름은 CommitAbility → GetCurrentAttackConfig → SetIsAttacking(true) → WaitGameplayEvent(Event.Combat.Confirm) + PlayMontageAndWait 순서로 진행됩니다. WaitGameplayEvent 태스크가 먼저 등록되어야 AnimNotify 이벤트를 놓치지 않습니다. OnHitConfirmEvent에서 UAO_MeleeHitEventPayload가 없으면 정면 방향 fallback으로 TraceStart/TraceEnd를 계산하는데, 이는 Payload 없는 단순 Notify에서도 히트 판정이 동작하도록 보장합니다. ApplyDamageAndKnockback에서 SetByCaller 방식으로 Data.Damage 태그에 데미지 값을 전달하는데, DamageEffectClass 내부에서 이 태그를 참조하는 Modifier가 정의되어야 합니다. 넉백 방향 계산에서 Z=0.2로 하드코딩한 뒤 Normalize하여 약간의 상승 벡터를 추가하고, SetMovementMode(MOVE_Falling)으로 강제 전환 후 LaunchCharacter를 호출합니다. SendHitReactEvent에서 KnockbackStrength >= 400.f 임계값으로 Knockdown/Light를 구분하는데, 이 임계값이 하드코딩되어 있어 밸런스 조정 시 주의가 필요합니다.
변수명타입지정자설명

TraceChannel TEnumAsByte<ECollisionChannel> EditDefaultsOnly SphereTrace에 사용하는 충돌 채널. 기본값 ECC_Pawn. ConvertToTraceType으로 변환하여 사용.
KnockdownHitReactTag FGameplayTag EditDefaultsOnly 넉다운 히트 리액트 태그. 생성자에서 Event.Combat.HitReact.Knockdown으로 초기화. KnockbackStrength >= 400.f 일 때 사용.
DefaultHitReactTag FGameplayTag EditDefaultsOnly 일반 히트 리액트 태그. 생성자에서 Event.Combat.HitReact.Light로 초기화. Werewolf_Attack에서 Heavy로 오버라이드됨.

HitSoundAttenuation TObjectPtr<USoundAttenuation> EditDefaultsOnly BPReadOnly 타격음 감쇠 설정. PlaySoundAtLocation에 전달됨.

CanActivateAbility() ▶
하는 일
부모 CanActivateAbility 체크 후, AvatarActor를 AAO_AICharacterBase로 캐스트하여 기절 상태(IsStunned)와 공격 중 상태(IsAttacking)를 확인합니다. 둘 중 하나라도 해당하면 false를 반환합니다.
심층 분석

ActivateAbility() ServerInitiated ▶
하는 일
CommitAbility 후 AI 캐릭터에서 GetCurrentAttackConfig()로 공격 설정을 로드합니다. SetIsAttacking(true)을 호출하고, WaitGameplayEvent(Event.Combat.Confirm) 태스크와 PlayMontageAndWait 태스크를 생성하여 활성화합니다.
심층 분석
중요한 실행 순서: WaitGameplayEvent 태스크를 먼저 ReadyForActivation한 후 PlayMontageAndWait를 활성화합니다. 이 순서가 역전되면 몽타주 초기 프레임의 AnimNotify 이벤트를 놓칠 수 있습니다. PlayMontageAndWait의 콜백으로 OnCompleted(정상 종료), OnCancelled(취소), OnInterrupted(인터럽트)를 각각 바인딩합니다. OnBlendOut은 바인딩하지 않아 블렌드 아웃 중에도 히트 판정이 계속 동작합니다. AttackMontage가 nullptr이면 즉시 EndAbility(bWasCancelled=true)로 종료합니다.
EndAbility() ▶
하는 일
AI 캐릭터의 SetIsAttacking(false)를 호출하여 공격 상태를 해제하고 부모 EndAbility()를 호출합니다.
심층 분석
AvatarActor 캐스트에 실패해도(캐릭터가 이미 파괴된 경우) 크래시하지 않고 Super::EndAbility를 호출합니다. GAS 태스크(WaitGameplayEvent, PlayMontageAndWait)는 부모 EndAbility에서 자동으로 정리됩니다.
OnHitConfirmEvent() ▶
하는 일
AnimNotify에서 전달된 Payload를 분석하여 SphereTrace 시작/종료 지점을 결정합니다. Payload에 UAO_MeleeHitEventPayload가 있으면 해당 정보를, 없으면 정면 방향으로 fallback합니다. SphereTraceMulti 결과에서 플레이어만 필터링하여 ApplyDamageAndKnockback을 호출합니다.
심층 분석
TSet<AActor*> ProcessedActors로 동일 액터 다중 히트를 방지합니다. SphereTraceMulti에서 EDrawDebugTrace::None을 사용하므로 디버그 시에는 코드를 변경해야 합니다. TraceRadius는 CurrentAttackConfig.AttackRadius를 사용하고, fallback 시 TraceEnd는 ActorForwardVector * AttackDistance로 계산됩니다. Cast<AAO_PlayerCharacter>로 플레이어만 필터링하여 AI 간 아군 공격(friendly fire)은 발생하지 않습니다.
ApplyDamageAndKnockback() virtual ▶
하는 일
무적(Status.Invulnerable) 확인 후, DamageEffectClass를 SetByCaller 방식으로 적용합니다. SendHitReactEvent를 호출하고, 넉백 방향 계산(Z=0.2) 후 MovementMode를 Falling으로 전환하여 LaunchCharacter를 실행합니다. 마지막으로 OnTargetHit 가상 함수를 호출합니다.
심층 분석
실행 순서: (1) 무적 체크 (2) 타격음(MulticastPlayHitSound) (3) GE 데미지 적용 (4) HitReact 이벤트 전송 (5) 넉백 물리 적용 (6) OnTargetHit 콜백. HitReact를 넉백보다 먼저 보내는 이유는 피격 상태에 진입시킨 후 물리적으로 날리기 위함입니다. MakeEffectContext에서 AddInstigator(InstigatorActor, InstigatorActor)로 Instigator와 Causer를 동일하게 설정하는데, 이는 Causer에 무기 액터를 넣지 않는 AI 특성입니다. Werewolf_Attack에서 이 함수를 오버라이드하여 자체 DamageEffectClass를 사용하고, 추가로 OnTargetHit에서 Heavy HitReact를 재전송합니다.
SendHitReactEvent() virtual ▶
하는 일
넉백 강도가 400 이상이면 KnockdownHitReactTag, 미만이면 DefaultHitReactTag를 선택하여 SendGameplayEventToActor로 타겟에게 HitReact 이벤트를 전송합니다.

심층 분석

역할

GAS 카테고리
Ability.State.Stunned · InstancedPerActor · Event.AI.Stunned 트리거
실행 위치
ServerInitiated — 서버에서 기절 상태를 관리하고 Effect 적용/해제
블록 태그
ActivationBlockedTags: Status.Debuff.Stunned — 기절 중 재기절 불가
Event.AI.Stunned 이벤트로 트리거되는 기절 처리 어빌리티입니다. 기절 시작 시 OnStunBegin()을 호출하고, 서버에서 이동을 비활성화(DisableMovement)하며, StunEffectClass를 적용하여 Status.Debuff.Stunned 태그를 부여합니다. 기절 해제 시 이동을 MOVE_Walking으로 복구하고, Effect를 제거하며, OnStunEnd()를 호출합니다. 기절 모션(StunMontage)이 없으면 DefaultStunDuration(기본 3초) 타이머로 자동 해제됩니다.
[심층 분석] 생성자에서 AbilityTriggers에 Event.AI.Stunned를 등록하고, AbilityTags에 Ability.State.Stunned를, ActivationBlockedTags에 Status.Debuff.Stunned를 설정합니다. 이 조합으로 (1) GameplayEvent로만 활성화 (2) 이미 기절 중이면 재기절 불가가 보장됩니다. HasAuthority 체크로 서버에서만 DisableMovement/Effect 적용을 수행하여 이동 복구와 Effect 제거가 Authority에서만 일어납니다. StunMontage가 nullptr인 경우 FTimerDelegate::CreateWeakLambda로 약한 참조 타이머를 사용하는데, 어빌리티가 먼저 종료되면 람다가 실행되지 않습니다. OnMontageCompleted에서 CurrentActorInfo nullptr 체크를 수행하여 비정상 종료 시 크래시를 방지합니다. MontageTask의 OnBlendOut도 OnMontageCompleted에 바인딩하여, 블렌드 아웃 시점에도 기절이 해제됩니다.
변수명타입지정자설명
StunMontage TObjectPtr<UAnimMontage> EditDefaultsOnly BPReadOnly 기절 모션. nullptr이면 DefaultStunDuration 타이머로 대체.
StunEffectClass TSubclassOf<UGameplayEffect> EditDefaultsOnly BPReadOnly 기절 중 적용할 GE. 내부에 Status.Debuff.Stunned 태그를 부여하는 Modifier가 필요.
DefaultStunDuration float EditDefaultsOnly BPReadOnly 기절 기본 지속 시간(초). 기본값 3.0. StunMontage가 없을 때만 사용.
StunEffectHandle FActiveGameplayEffectHandle (private) 적용된 기절 Effect 핸들. EndAbility에서 RemoveActiveGameplayEffect에 사용.

심층 분석

UAO_GA_Bull_Charge

역할
Bull 돌진 공격 어빌리티 — 전방 SphereTrace로 적 감지 후 넉다운 적용
GAS 카테고리
InstancedPerActor · ServerInitiated
실행 위치
ServerInitiated — 서버에서 돌진 상태 관리 및 히트 판정
설계 패턴

UAO_EQS_Context_Target
부모: UEnvQueryContext
역할
AI의 추격 대상(플레이어)을 EQS Context로 제공
주요 용도
추격 대상 근처/주변 위치 검색, 포위 위치 계산
데이터 소스
AAO_AggressiveAICtrl::GetChaseTarget()
Owner 처리
Pawn(GetController) 또는 Controller 직접 캐스트
QueryOwner에서 AAO_AggressiveAICtrl을 획득하고, GetChaseTarget()으로 추격 대상 플레이어를 가져와 단일 Actor Context로 제공합니다. 대상이 없으면 컨텍스트를 설정하지 않습니다.
[심층 분석] PackMembers Context와 동일한 Pawn/Controller 양방향 패턴을 사용합니다. 단일 Actor를 SetContextHelper에 전달하여 "이 Target 근처"같은 EQS Generator의 중심점으로 사용됩니다. AggressiveAICtrl에 의존하므로, 다른 타입의 AI Controller를 사용하는 적은 이 Context를 사용할 수 없습니다. ChaseTarget이 nullptr이면 SetContextHelper가 호출되지 않아 EQS 쿼리 결과가 비어 있게 됩니다.

(자체 멤버 변수 없음)

UAO_EQS_Test_LineOfSight
부모: UEnvQueryTest
역할
모든 플레이어의 시야에서 보이지 않는 위치인지 체크하는 EQS Test
비용
EEnvTestCost::Medium — 플레이어 수 x 아이템 수 LineTrace 수행
결과 타입
Bool (float 아님) — 모든 플레이어가 못 보면 1.0, 아니면 0.0
주요 용도
적 스폰 위치 선정 시 플레이어 시야 밖 필터링 (Lethal Company/REPO 스타일)
모든 플레이어의 눈 위치(PlayerEyeHeightOffset)에서 각 EQS 아이템(SpawnLocationHeightOffset 적용)으로 LineTraceSingleByChannel을 수행합니다. 하나라도 막히지 않으면(플레이어가 볼 수 있으면) 해당 아이템은 실패(0.0)합니다. HitResult의 Distance가 전체 거리의 95% 이상이면 "거의 막히지 않음"으로 판정하여 볼 수 있는 것으로 처리합니다.
[심층 분석] SetWorkOnFloatValues(false)로 bool 모드에서 동작합니다. ValidItemType이 StaticClass()(가장 기본)로 설정되어 모든 EQS 아이템 타입에서 동작합니다. LineTrace 최적화: 한 플레이어라도 볼 수 있으면 즉시 break하여 불필요한 트레이스를 건너뜁니다. 95% 거리 임계값(0.95f)은 Trace가 아이템 위치 근처에서 끝나는 경우(예: 아이템이 벽 바로 뒤에 있을 때)를 "보이는 것"으로 처리합니다. 이 로직은 FVector::Dist(PlayerEyeLocation, ItemLocation) * 0.95f와 HitResult.Distance를 비교합니다. CollisionQueryParams에서 Player와 QueryOwner(AI)를 IgnoredActor로 추가하여 자기 자신과의 충돌을 방지합니다. 플레이어가 0명이면 모든 아이템을 1.0(통과)으로 설정하여, 플레이어가 없는 상태에서도 쿼리가 결과를 반환합니다. 비용이 Medium으로 선언되어 EQS 최적화에서 다른 Low 비용 테스트보다 나중에 실행됩니다.
변수명타입지정자설명
PlayerEyeHeightOffsetfloatEditDefaultsOnly플레이어 눈 높이 오프셋. 기본값 160.
SpawnLocationHeightOffsetfloatEditDefaultsOnly스폰 위치 높이 오프셋. 기본값 100.
TraceChannelTEnumAsByte<ECollisionChannel>EditDefaultsOnlyLine Trace 채널. 기본값 ECC_Visibility.
RunTest() ▶
하는 일
GetAllActorsOfClass로 모든 플레이어 수집 → 각 아이템에 대해 모든 플레이어의 눈 위치에서 LineTraceSingleByChannel → 하나라도 !bHit 또는 Distance > 95%이면 0.0, 전부 막히면 1.0.
심층 분석
O(P * I) 복잡도 (P=플레이어 수, I=아이템 수). 3인 코옵에서 100개 아이템이면 최대 300회 LineTrace이므로 Medium 비용이 적절합니다. Early-out(bAllPlayersCannotSee=false 시 break)으로 최악 케이스를 줄입니다.

UAO_EQS_Test_NavAreaExclusion
부모: UEnvQueryTest
역할
특정 NavArea(스폰 금지 영역)와의 겹침을 체크하는 EQS Test
비용
EEnvTestCost::Low
기본 제외 영역
UAO_NavArea_SpawnRestriction (커스텀 NavArea)
내부 동작
NavModifierVolume의 AreaClass 매칭 + EncompassesPoint 체크
NavigationSystemV1에서 NavMesh를 조회하고, 모든 NavModifierVolume을 순회하여 RestrictedNavAreaClass와 일치하는 볼륨을 찾습니다. 각 EQS 아이템에 대해 해당 볼륨의 EncompassesPoint를 체크하여 내부면 0.0, 외부면 1.0을 설정합니다.
[심층 분석] 먼저 NavMesh에 ProjectPointToNavigation으로 아이템을 프로젝션하고, 프로젝션 실패 시 즉시 0.0으로 처리합니다. 이는 NavMesh가 없는 영역을 자동으로 제외합니다. GetAllActorsOfClass(ANavModifierVolume)로 모든 NavModifier 볼륨을 순회하는데, 볼륨 수가 많으면 성능에 영향을 줄 수 있습니다. 그러나 Cost=Low로 선언되어 EQS 최적화에서 먼저 실행되어 다른 테스트의 아이템 수를 줄입니다. RestrictedNavAreaClass 기본값이 UAO_NavArea_SpawnRestriction::StaticClass()로 생성자에서 설정됩니다. NavSys가 nullptr이면 모든 위치를 통과시키는 안전 처리가 있습니다.
변수명타입지정자설명
RestrictedNavAreaClassTSubclassOf<UNavArea>EditDefaultsOnly제외할 NavArea 클래스. 기본값 UAO_NavArea_SpawnRestriction.
RunTest() ▶
하는 일
NavSystem 획득 → 각 아이템을 NavMesh에 프로젝션 → GetAllActorsOfClass(ANavModifierVolume)로 모든 볼륨 순회 → RestrictedNavAreaClass와 AreaClass 일치 + EncompassesPoint 체크 → 내부면 0.0, 외부면 1.0.
심층 분석
ProjectPointToNavigation의 extent가 FVector(500, 500, 200)으로 꽤 넓어, 수평 500 단위 내에서 가장 가까운 NavMesh 포인트를 찾습니다. NavModifierVolume::GetAreaClass()로 볼륨의 NavArea 타입을 확인하고, 일치하는 볼륨이 발견되면 즉시 break하여 불필요한 순회를 피합니다.
방향대상 클래스관계 설명

UAO_EQS_Test_PlayerFOV
부모: UEnvQueryTest
역할
모든 플레이어의 시야각(FOV) 밖에 있는지 체크하는 EQS Test
비용
EEnvTestCost::Low — LineTrace 없이 DotProduct만 사용
결과 타입
Bool — 모든 플레이어 FOV 밖이면 1.0, 아니면 0.0
카메라 지원
bUseCameraDirection=true면 PC의 ControlRotation 사용
각 플레이어의 시야 방향(카메라 또는 캐릭터 Forward)과 아이템 방향의 DotProduct를 FOV 코사인과 비교합니다. MinDistanceFromPlayer 이내면 FOV와 무관하게 실패하여 바로 옆/뒤 스폰을 방지합니다.
[심층 분석] FOV 코사인을 FMath::Cos(FMath::DegreesToRadians(PlayerFOVDegrees * 0.5f))로 미리 계산하여 루프 내에서의 삼각함수 호출을 방지합니다. bUseCameraDirection이 true이면 PC->GetControlRotation().Vector()를 사용합니다. 3인칭 게임에서 캐릭터 Forward와 카메라 방향이 다를 수 있으므로 이 옵션이 중요합니다. MinDistanceFromPlayer(기본 500)는 뒤에서 스폰되는 것을 방지합니다 — FOV 밖이지만 플레이어 바로 뒤인 경우를 차단합니다. LineOfSight 테스트와 달리 물리적 차단(벽 등)을 고려하지 않으므로, 두 테스트를 조합하면 "시야 밖 + 시선 차단" 이중 필터링이 가능합니다. DotProduct >= FOVCosine 비교에서 등호를 포함하여 정확히 시야각 경계에 있는 위치도 "보이는 것"으로 처리합니다.
변수명타입지정자설명
PlayerFOVDegreesfloatEditDefaultsOnly플레이어 시야각(도). 기본값 90. 절반 각도(45도)로 계산.
MinDistanceFromPlayerfloatEditDefaultsOnly최소 거리. 기본값 500. 이 이내면 FOV 무관 실패.
bUseCameraDirectionboolEditDefaultsOnly카메라 방향 사용 여부. 기본값 true.
RunTest() ▶
하는 일
FOV 코사인 미리 계산 → 각 아이템에 대해 모든 플레이어와 (1) MinDistance 체크 (2) Forward/Camera 방향과 DotProduct 비교 → FOV 내부면 0.0, 모두 밖이면 1.0.
심층 분석
비용이 Low로 선언되어 LineOfSight(Medium)보다 먼저 실행됩니다. FOV 테스트로 아이템을 먼저 걸러낸 후 LineTrace를 수행하면 전체 성능이 향상됩니다. 이 순서는 EQS 프레임워크가 Cost 기반으로 자동 최적화합니다.

UAO_EQS_Test_VolumeExclusion
부모: UEnvQueryTest
역할
스폰 금지 볼륨(AAO_Area_SpawnRestriction)과의 겹침을 체크하는 EQS Test
비용
EEnvTestCost::Low
결과 타입
Bool — 금지 볼륨 밖이면 1.0, 안이면 0.0
NavArea 대비
NavMesh 의존 없이 커스텀 볼륨으로 직접 겹침 체크
RestrictionVolumeClass(기본 AAO_Area_SpawnRestriction)로 GetAllActorsOfClass를 수행합니다. 각 EQS 아이템에 대해 모든 금지 볼륨의 EncompassesPoint를 체크하여, 금지 영역 안이면 0.0, 밖이면 1.0으로 점수를 설정합니다.
[심층 분석] NavAreaExclusion과 구조가 유사하지만 핵심 차이가 있습니다: (1) NavMesh 프로젝션 없음 (2) NavModifierVolume 대신 커스텀 AAO_Area_SpawnRestriction 사용 (3) NavigationSystem 의존 없음. RestrictionVolumeClass가 nullptr이면 AAO_Area_SpawnRestriction::StaticClass()를 fallback으로 사용하여, 에디터에서 설정하지 않아도 기본 동작합니다. 두 테스트(NavAreaExclusion + VolumeExclusion)를 조합하면 NavMesh 기반 + 커스텀 볼륨 기반 이중 스폰 제한이 가능합니다. 금지 볼륨이 0개이면 모든 아이템을 통과시키는 안전 처리가 있습니다.
변수명타입지정자설명
RestrictionVolumeClassTSubclassOf<AAO_Area_SpawnRestriction>EditDefaultsOnly테스트할 스폰 금지 볼륨 클래스. 미설정 시 AAO_Area_SpawnRestriction 사용.
